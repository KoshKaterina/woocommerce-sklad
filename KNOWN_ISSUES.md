# Известные особенности и потенциальные проблемы

## Статус заказа (state) — href в meta-ссылке

`make_meta("state", uuid)` генерирует href вида:
```
.../entity/state/{uuid}
```

По документации API МойСклад, корректный href для статуса заказа покупателя:
```
.../entity/customerorder/metadata/states/{uuid}
```

**Текущее поведение:** статусы устанавливаются корректно — предположительно потому, что используемый статус является первым (дефолтным) в списке МойСклад, и API резолвит по UUID.

**Риск:** если будет использован не-дефолтный статус, или API МойСклад ужесточит валидацию href — установка статуса перестанет работать.

**Исправление при необходимости:** собирать meta для state вручную:
```python
body["state"] = {
    "meta": {
        "href": f"{base_url}/entity/customerorder/metadata/states/{state_uuid}",
        "type": "state",
        "mediaType": "application/json",
    }
}
```

## InSales: отложенные задачи

### Вебхук InSales — нет HMAC-верификации
InSales не поддерживает подпись вебхуков. При реализации эндпоинта `/webhook/insales/order` нужно верифицировать данные через дозапрос заказа по API (`get_order(id)`) перед обработкой, иначе любой может создать фейковый заказ в МС.

### Reconciliation InSales
- Polling каждые 3 минуты через `updated_since` + `from_id`
- Холодный старт: дефолтное окно = 1 час (не грузить все заказы)
- Два прохода: создание пропущенных + маркировка оплаты по `paid_at`/`financial_status`
- Нужен общий lock с WC reconciliation для защиты от конкурентного создания контрагентов/заказов

### Коллизия номеров заказов между источниками
Все три источника (WC, InSales, uCoz) используют числовые id, которые могут совпасть. Защита — постоянные суффиксы: WC без суффикса, InSales " Tangemshop", uCoz " TGShop". Источник = суффикс. Снимать нельзя.

### Дубликат услуги "Наценка за наложенный платеж"
В МС существуют 2 услуги с одинаковым названием. `find_or_create_service` находит первую. Нужно убедиться, что это правильная услуга, или удалить дубликат.

### Промокод WC — проверить поле API
`extract_promo_code()` читает `order_data.get("coupon_codes")`. В WC REST API купоны могут быть в `coupon_lines[].code`. Нужно проверить реальный формат и исправить при необходимости.

### Маркировка оплаты InSales
Проверять `paid_at is not None` AND `financial_status == "paid"`. Для COD — не маркировать. Реализовать в reconciliation (polling) + вебхук как ускоритель.

### Пагинация InSales: стоп-условие по пустой странице
`InSalesClient.get_orders` сейчас выходит из цикла при пустом `batch` (`if not batch: break`). Это даёт +1 лишний запрос к API на каждый прогон и — теоретически — бесконечный цикл, если API по `from_id` возвращает `id >= from_id` (а не строго `>`). Сейчас на практике работает. Безопаснее добавить `if len(batch) < per_page: break` после `extend`, но до подтверждения поведения API трогать не будем.
