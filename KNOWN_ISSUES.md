# Известные особенности и потенциальные проблемы

## Статус заказа (state) — href в meta-ссылке

`make_meta("state", uuid)` генерирует href вида:
```
.../entity/state/{uuid}
```

По документации API МойСклад, корректный href для статуса заказа покупателя:
```
.../entity/customerorder/metadata/states/{uuid}
```

**Текущее поведение:** статусы устанавливаются корректно — предположительно потому, что используемый статус является первым (дефолтным) в списке МойСклад, и API резолвит по UUID.

**Риск:** если будет использован не-дефолтный статус, или API МойСклад ужесточит валидацию href — установка статуса перестанет работать.

**Исправление при необходимости:** собирать meta для state вручную:
```python
body["state"] = {
    "meta": {
        "href": f"{base_url}/entity/customerorder/metadata/states/{state_uuid}",
        "type": "state",
        "mediaType": "application/json",
    }
}
```

## InSales

### Вебхук InSales — нет HMAC-верификации (отложено)
Вебхука InSales нет (заказы идут через сверку). Если делать `/webhook/insales/order`:
InSales не подписывает вебхуки → верифицировать дозапросом по API (`get_order(id)`),
иначе любой создаст фейковый заказ в МС.

### Reconciliation InSales — реализовано
Сверка раз в 3 мин (окно 9 мин): `updated_since` + клиентский фильтр верхней границы;
создание пропущенных + маркировка оплаты (`paid_at` + `financial_status=="paid"`, COD не маркируется).
**Открыто:** нет общего lock между сверкой (Timer-поток) и WC-вебхуком → теоретическая
гонка при конкурентном создании контрагента/заказа.

### Бэклог при включении InSales
Сверка берёт только заказы, изменённые за последние 9 мин. Заказы, накопившиеся до
включения адаптера, сами не подтянутся — нужен разовый бэкфилл (по номерам/датам).

### Коллизия номеров заказов между источниками
Все три источника (WC, InSales, uCoz) используют числовые id, которые могут совпасть. Защита — постоянные суффиксы: WC без суффикса, InSales " Tangemshop", uCoz " TGShop". Источник = суффикс. Снимать нельзя.

### Дубликат услуги "Наценка за наложенный платеж"
В МС существуют 2 услуги с одинаковым названием. `find_or_create_service` находит первую. Нужно убедиться, что это правильная услуга, или удалить дубликат.

### Промокод WC — проверить поле API
`extract_promo_code()` читает `order_data.get("coupon_codes")`. В WC REST API купоны могут быть в `coupon_lines[].code`. Нужно проверить реальный формат и исправить при необходимости.

### Пагинация InSales: стоп-условие по пустой странице
`InSalesClient.get_orders` сейчас выходит из цикла при пустом `batch` (`if not batch: break`). Это даёт +1 лишний запрос к API на каждый прогон и — теоретически — бесконечный цикл, если API по `from_id` возвращает `id >= from_id` (а не строго `>`). Сейчас на практике работает. Безопаснее добавить `if len(batch) < per_page: break` после `extend`, но до подтверждения поведения API трогать не будем.

## Адрес доставки (shipmentAddressFull)

### Страна тихо опускается при несовпадении имени
`ms_client.find_country_meta` ищет страну в справочнике МС по точному имени (`filter=name=...`),
которое берётся из таблицы `ISO_TO_COUNTRY_NAME` (ISO `shipping.country` → имя). Если в справочнике
МС страна названа иначе (напр. «Российская Федерация» вместо «Россия») или ISO-кода нет в таблице —
`country` в `shipmentAddressFull` **не пишется** (остальные поля — пишутся). Это намеренный graceful
fallback, но при появлении новых стран проверить имена в справочнике МС и при необходимости расширить
`ISO_TO_COUNTRY_NAME`.

### Парсер целится на чистый DaData-формат
`core/address_parser` рассчитан на стандартизованную плагином DaData строку `shipping.address_1`
(стандартизация на checkout обязательна). На «грязном» ручном вводе (легаси-заказы до DaData) разбор
улицы/дома/квартиры может быть неточным — но такие заказы не переобрабатываются (адрес пишется только
при создании). Регион (справочник МС) не пишется; «Другое» (addInfo) не заполняем вовсе —
`shipping.state` либо дублирует город, либо грязный (аудит 2026-06-11: дубль в 57/60 заказов,
вычищено ретро-скриптом). ПВЗ — best-effort из `shipping.address_2`. InSales разнесение адреса
пока не реализовано (только плоский `shipmentAddress`).

### address_2 для СДЭК-ПВЗ ненадёжен — код ПВЗ берём из меты
Чекаут иногда не записывает адрес ПВЗ в `shipping.address_2` (реальный сбой: заказ 17130 —
1 из 42). Код ПВЗ поэтому читается в первую очередь из меты официального CDEK-плагина
`shipping_lines[0].meta_data._official_cdek_office_code` (есть всегда), regex по `address_2` —
fallback. Город — fallback из `_official_cdek_city`. Улицу ПВЗ при пустом `address_2` взять
неоткуда (CDEK API сознательно не подключаем) — в МС уйдут код+город, плоский адрес
«Россия, <код>, <город>».

### МС перетирает плоский shipmentAddress при наличии shipmentAddressFull
Когда передан `shipmentAddressFull`, МС генерирует отображаемую плоскую строку из его
компонентов, игнорируя наш `shipmentAddress`. Качество full = качество отображения. И наоборот:
если писать ТОЛЬКО плоский адрес, МС кладёт всю строку в `shipmentAddressFull.addInfo`
(так выглядят заказы, созданные до фичи разнесения).

### Чекаут «Доставка курьером по Москве» не передаёт адрес (регрессия сайта после DaData)
Заказы 17083, 17109 (после 08.06): адреса нет НИГДЕ в payload WC (`shipping`, `billing`, мета,
комментарий — проверено побайтово по всему JSON). До подключения DaData поле заполнялось ручным
вводом в 40/45 заказов этого метода; после — 0/2. Похоже, DaData-виджет подменил стандартное поле
и не привязан к методу `flat_rate` (instance_id=17). Интеграция бессильна — чинить надо в чекауте
сайта. В МС уходит только город; адрес менеджер узнаёт у клиента.

### Заказы из amoCRM-виджета: весь адрес в «Коде ПВЗ»
Заказы, созданные виджетом amoCRM (признаки: «Заказ создан через виджет», «Ссылка на сделку»,
НЕТ «Номера заказа на сайте»), кладут всю строку адреса ПВЗ в доп.поле «Код ПВЗ»
(напр. 04277: «PRM35, Пермь, пр-т Парковый, 23»), а «Адрес доставки» не заполняют. Это поведение
amoCRM-виджета, НЕ нашей интеграции (наша пишет только код, напр. «PRM35»); существует минимум
с 19.05.2026 (03488). ~5 заказов на 1200. Чинить — на стороне настройки amo-виджета, если мешает.
